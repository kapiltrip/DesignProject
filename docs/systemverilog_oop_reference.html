SystemVerilog OOP Concepts

Quick reference for verification testbenches: inheritance, polymorphism, abstraction, and encapsulation.

Big idea: In a SystemVerilog testbench, classes help you model verification components such as transactions, generator, driver, monitor, scoreboard, and environment. OOP is mainly about organizing behavior, reusing code, and controlling how components interact.

One-Line Revision

Concept Meaning Main SV Keywords
Inheritance Build a child class from a parent class so the child can reuse and extend existing data and methods. extends, super
Polymorphism Use a base-class handle to call a method, but execute the child class version at run time. virtual function, virtual task
Abstraction Define what behavior is required while leaving details to a derived class. virtual class, pure virtual
Encapsulation Protect internal data and expose controlled methods to read or modify it. local, protected

1. Inheritance

Definition: Inheritance means a child class is created from a parent class. The child class gets the parent class members and can add new members or override behavior.

Meaning in verification: Use inheritance when several components share common behavior but each one needs a small specialization. For example, a base transaction can hold common fields, and a child transaction can add protocol-specific fields.

class base_txn;
  bit [7:0] data;

  function void print();
    $display("data=%0d", data);
  endfunction
endclass

class fifo_txn extends base_txn;
  bit rd_en;
  bit wr_en;
endclass

Here, fifo_txn inherits data and print() from base_txn, then adds rd_en and wr_en.

2. Polymorphism

Definition: Polymorphism means "many forms." In SystemVerilog, a parent-class handle can point to a child-class object. If the method is declared virtual, the simulator calls the child version of that method.

Meaning in verification: A testbench can call the same method name on different child objects and get different behavior. This is useful for reusable drivers, sequences, and tests.

class base_test;
  virtual task run();
    $display("base run");
  endtask
endclass

class fifo_test extends base_test;
  task run();
    $display("fifo run");
  endtask
endclass

module tb;
  base_test t;
  fifo_test f;

  initial begin
    f = new();
    t = f;
    t.run(); // prints: fifo run
  end
endmodule

The handle type is base_test, but the object is fifo_test. Because run() is virtual in the base class, the child implementation runs.

Important: Without virtual on the base method, a base-class handle may call the base method instead of the child method. For polymorphism, mark the base method as virtual.

3. Abstraction

Definition: Abstraction means the base class describes the required behavior, but does not fully implement it. In SystemVerilog this is commonly done with virtual class and pure virtual methods.

Meaning in verification: Use abstraction when every child component must provide a certain method, but each child implements it differently.

virtual class base_driver;
  pure virtual task drive();
endclass

class fifo_driver extends base_driver;
  task drive();
    $display("drive FIFO signals");
  endtask
endclass

base_driver cannot be constructed directly because it is a virtual class. It says every real driver must provide drive(). The child class fifo_driver provides the actual code.

4. Encapsulation

Definition: Encapsulation means keeping internal class data protected and allowing access only through controlled methods.

Meaning in verification: Use encapsulation to prevent accidental changes to important state, such as error counts, transaction IDs, or scoreboard queues.

class scoreboard;
  local int err_count;

  function void add_error();
    err_count++;
  endfunction

  function int get_error_count();
    return err_count;
  endfunction
endclass

Because err_count is local, code outside the class cannot directly change it. Other code must call add_error() or get_error_count().

local vs protected

Qualifier Who Can Access It?
local Only methods inside the same class.
protected Methods inside the same class and methods inside derived child classes.
No qualifier Public by default in SystemVerilog class members.

Abstraction vs Encapsulation

Concept Focus Simple Memory Hook
Abstraction Hide or postpone implementation details by exposing only the required behavior. What should this class do?
Encapsulation Hide or protect internal data and control who can access it. Who can touch this data?

Abstraction Example

A base driver says every child driver must have drive(), but the base driver does not know the exact protocol signals.

Encapsulation Example

A scoreboard keeps its expected-data queue private so only scoreboard methods can push, pop, or clear it.

How This Relates To Your FIFO Testbench

Class OOP Role
transaction Represents one operation or observed activity. It carries fields like oper, wr_data, rd_data, full, and empty.
generator Creates randomized transaction objects and sends them to the driver.
driver Converts transaction fields into pin-level FIFO activity on the virtual interface.
monitor Samples actual FIFO interface signals and sends observed transaction objects to the scoreboard.
scoreboard Stores expected write data and compares expected data against actual rd_data.
environment Connects all components together and controls test phases.

Your current testbench uses classes, objects, mailboxes, and virtual interfaces, but it does not yet heavily use inheritance, polymorphism, abstraction, or access qualifiers. Those concepts become more useful as the testbench grows into reusable base classes and child classes.

Short Exam-Style Definitions

  • Inheritance: Mechanism to create a derived class from a base class using extends, allowing reuse and extension of properties and methods.
  • Polymorphism: Mechanism where a base-class handle can refer to a derived-class object and a virtual method call executes the derived implementation.
  • Abstraction: Mechanism to specify required behavior without exposing or providing all implementation details, often using virtual class and pure virtual methods.
  • Encapsulation: Mechanism to protect class data and control access to it using class methods and access qualifiers such as local and protected.
